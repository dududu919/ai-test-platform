# AI 测试平台（本地 MVP）产品设计文档

## 1. 产品定位
这是一个面向“简单业务系统”的本地测试平台 MVP，用最少组件实现一个可演示、可验证的 AI 测试闭环。

核心目标：
用结构化场景驱动自动化测试，并在失败时给出可读的原因与建议。

非目标：
不做多 agent 编排、不做全自动自愈、不做生产环境探索式测试。

## 2. 目标用户
1. 需要快速验证测试流程价值的产品/测试负责人  
2. 想展示 AI 测试闭环能力的技术团队  
3. 需要低成本 demo 的内部验证场景

## 3. MVP 能力边界
当前 MVP 只覆盖以下能力：
1. 结构化场景生成  
2. Playwright 执行  
3. 失败证据采集  
4. 失败分类与报告输出

不覆盖：
1. 视觉异常检测  
2. 全自动修复  
3. 自动缺陷创建  
4. 生产环境探索

## 4. 核心概念
### 4.1 结构化场景（Scenario）
结构化场景是平台的核心中间格式。  
它将“业务流程”抽象为可执行步骤和可验证断言。

示例：
```json
{
  "id": "users-create-success",
  "preconditions": ["管理员先登录"],
  "steps": [
    "goto /login",
    "fill username",
    "fill password",
    "click login",
    "goto /users/new",
    "click submit"
  ],
  "assertions": ["页面出现：用户创建成功"]
}
```

### 4.2 失败分类
MVP 只做 5 类失败：
1. 元素定位失败  
2. 断言失败  
3. 权限失败  
4. 后端接口异常  
5. 数据未正确落库

## 5. 核心流程
1. 读取配置与代码结构  
2. 生成结构化测试场景  
3. 编译并执行 Playwright  
4. 采集日志 / 截图 / 网络摘要  
5. 失败分类与报告输出

## 6. 产品界面（控制台）
控制台的目标是“让演示者不需要命令行”。  
它应包含：

1. 被测网站状态  
2. 测试流程状态  
3. 最新报告入口  
4. 场景数量  
5. 一键运行测试

## 7. 成功指标
MVP 不追求“生成用例数量”。  
衡量指标：
1. 核心流程覆盖率提升  
2. 手写测试时间下降  
3. 失败报告可读性提升  

## 8. 演示闭环
最小闭环展示包含：
1. 本地 demo 网站  
2. 测试平台控制台  
3. 自动生成场景  
4. Playwright 执行  
5. 失败报告页面

## 9. 后续演进方向
1. 多系统类型适配（CRM / 审批 / 订单 / CMS）  
2. 自定义场景模板  
3. 更丰富的断言（API / DB / 状态流转）  
4. 报告页面增强（对比、趋势、可搜索）
